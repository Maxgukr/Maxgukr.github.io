BIOS启动过程

计算机加电后，最先运行的不是操作系统,　而是执行系统初始化软件完成基本IO初始化和引导加载功能。简单的说，系统初始化软件就是在操作系统运行之前运行的一小段程序。通过这一小段程序，我们可以初始化硬件设备等操作，为后续加载内核到内存做好准备。之后引导程序会从外部存储设备的主分区读取操作系统内核印象加载到内存中，并将控制权交给系统内核。

从上述可以知道，操作系统进到内存控制计算机的运行需要两个帮手来帮助他，一个叫BIOS，一个叫bootloader。BIOS又称为固件，固化在内存中的ROM中，这是一种在掉电后可以继续保存的存储介质。计算机开机后，会初始化一些寄存器的值，其中CS和EIP被初始化为0xf000和0xfff0,CS的值左移４位加上EIP的值构成了上电后cpu索要执行的第一条指令，一般是一条长跳转指令，跳转到BIOS的程序的入口地址,开始执行BIOS。

BIOS 加载存储设备(比如软盘、硬盘、光盘、USB盘)上的第一个 扇区(主引导扇区,Master Boot Record,or MBR) 的512字节到内存的 0x7c00，然后跳转到ox7c00的第一条指令执行,接下来就是bootloader所要做的工作。

bootloader启动过程

BIOS将通过读取硬盘主引导分区到内存，并跳转到对应内存中的位置执行bootloader，bootloader执行的工作如下:

  • 切换到保护模式，启用分段机制
  • 读磁盘中ELF文件格式的操作系统到内存
  • 显示字符串信息
  • 把控制权交给操作系统

bootloader通过读取硬盘区间的ELF文件加载操作系统到内存后，就跳转到操作系统在内存中的入口位置，之后操作系统接管整个控制权。

地址空间

线性地址空间：一台计算机只有一个物理空间，但在操作系统的管理下，每个程序都认为自己独占整个计算机的物理空间。为了让多个程序有效的互相隔离和使用物理地址空间，引入线性地址空间，又叫虚拟地址空间

物理地址空间：从操作系统的角度看，它所要管理的内存就是一个大数组，物理地址空间就是这个大数组的索引值

muduo源码学习-buffer类

Posted on 2019-06-25 | Edited on 2019-09-15

本章节介绍muduo网络库中输入输出缓冲区的设计与实现。

Muduo的IO模型

在之前的文章《unix-io》中介绍了五种io模型：

  • 阻塞
  • 非阻塞
  • io复用
  • 信号驱动
  • 异步

上述都是单线程下的io模型。在C10K问题中,把线程也纳入了考量。在这个多核时代，线程是不可避免的。libevent的作者对服务端网络编程的线程模型有一个观点，one loop per thread is usually a good model.

如果采用one loop per thread的模型，那么多线程服务端的问题就简化为如何设计一个高效且易于使用的event loop,然后在每个线程run一个event loop就可以了（同步和互斥是不可或缺的）。

event loop是non-blocking网络编程的核心，在现实生活中，non-blocking几乎总是和IO multiplexing一起使用，原因有两点：

  • 用轮询(polling)来检查某个non-blocking IO操作是否完成，这样太浪费cpu时间
  • IO复用不能和阻塞IO一起使用，因为在阻塞IO中，read()/write()/accept()/connect()都有可能阻塞IO线程,这样线程就没办法处理其他socket上的事件

为什么在non-blocking编程中buffer是必须的

non-blocking的核心思想是避免线程阻塞在read()/write()或其他IO系统的调用函数上，这样可以最大限度的复用thread-of-control，让一个线程服务于多个socket连接.IO线程只能阻塞在IO复用函数上，像select/poll/epoll_wait这种。这样一来，缓冲层是必须的，每一个tcp连接都要有一个stateful的input buffer和output buffer。

TCP连接必须要有output buffer

TCP的发送缓冲区容量是有限的，由于流量控制的原因，上一次的数据还没发送完，新的数据又来了，但此时缓冲区已经放不下所有的待发送数据，程序肯定不能原地等待tcp发送完毕腾出空间接受新数据。作为应用层，只负责向操作系统生成数据，至于数据怎么发，分多少次发，何时发，这不是应用程序需要考虑的事情，这些工作应该交给网络库来完成。网络库应该接管剩余的不能放在TCP缓冲区里的数据，并注册EPOLLOUT事件，

  • 一旦socket变得可写，就立刻发送数据。
  • 如果任然不能一次发送完，需要持续关注EPOLLOUT事件；
  • 如果写完了，网络库应该停止关注EPOLLOUT事件。
  • 如果新到数据时，buffer中还有剩余，则应该append到buffer后面，而不是立即发送
  • 如果tcp关闭了连接，但buffer中还有剩余待发送数据，怎应等待数据发送完毕后在关闭连接

综上，要想tcp连接在buffer上不阻塞，必须为每个tcp连接创建output buffer

TCP连接必须要有input buffer

网络库在处理socket可读事件的时候，必须一次性把socket里面的数据读完(从操作系统缓冲区搬运到应用层buffer)，否则会反复触发EPOLLIN事件,造成busy-loop。上述要求必然造成数据接受不完整的情况，比如连续读取好几个socket的内容才能得到一个完整的消息。

Buffer的功能需求

muduo buffer的特点:

  • 对外表现为一段连续内存
  • size()可以自动增长，不是固定size()
  • 内部以std::vector来保存数据，并提供相应的访问函数
  • buffer像一个queue，从末尾写入数据，从头读出数据

在非阻塞网络编程中，如何设计并使用缓冲区？一方面需要减少系统调用，一次读的数据越多越好，似乎应该准备一个大的缓冲区。另一方面希望减少内存占用。如果有1000个并发连接，为每个连接准备50kb的读写内存的话，将占用1G的内存。所以，看似是个很矛盾的选择

muduo的做法是在栈上开辟一块零时的extrabuf空间,然后利用readv()来读取数据。buffer不够了就把数据先放到extrabuf里面，发送一部分之后再append回去。

libevent-2-event_loop.md

Posted on 2019-06-10

libevent的基本操作单元是事件。每个事件代表一个条件集合，包含:

  • 一个文件描述符准备好读或写
  • 一个文件描述符变得准备好读或写(边沿触发型的IO)
  • timeout事件
  • 信号量事件
  • 用户触发事件

事件都有相同的行为风格，一旦你调用一个事件函数去设置一个事件并关联到一个event_base，它就已经初始化了。当一个事件处于挂起状态的时候，如果触发一个事件的条件发生了，比如一个文件描述符的状态发生改变或者超时到期，事件就会由挂起转为激活状态，对应的回调函数(用户提供)就会执行执行。如果事件被设置为persistent，那么将继续维持挂起状态。

构建事件对象

为了创建一个新的事件，使用event_new()接口

event_new()函数分配和创建一个新的事件对象来给event_base结构体使用。what参数代表上面定义的标志位。fd表示的文件描述符是非负的，将可以观察到对应的读或写事件。当这个事件处于激活状态，将会调用对应的回调函数cb。arg是会传递给cb函数的参数。

一个新的事件被初始化为非挂起状态，想使他变为挂起状态，则应调用event_add()函数

Reactor的核心部分及其相关类实现

在前面的文章，高性能服务器框架中介绍了，Reactor是一种高效的服务端编程框架，通过可以轻松实现高并发。reactor一种有的三个主要部分：事件源、多路事件分发器、reactor反应器。其中多路事件分发器是最核心的部分。以下是类在reactor中扮演的角色。

  • Channel类 多路事件分发器。每个Chaannel对象自始至终都只属于一个EvnentLoop，因此每个Channel对像都属于某一个IO线程。每个Channel对象自始至终只负责一个文件描述符(事件源，fd)的IO事件分发，但它并不拥有这个fd，也不会在析构的时候关闭这个fd.　Channel会把不同的IO事件生成不同的回调，例如ReadCallBack,WriteCallBack.

  • Poller类　IO复用机制的封装。Poller是基类，会派生出两个子类：EPollPoller和PollPoller,分别对应poll(2)和epoll(4)两种多路复用机制。poller在EventLoop中作为成员变量使用，只提供给所在事件循环的io线程中使用。

  • EventLoop类　Reactor的核心类。EventLoop可以看作是Reactor中的指挥官，通过poller_成员监听事件，通过channel类来调用相关的回调函数执行相关操作。

运行一个事件循环

一旦你有了一个event_base并注册了一些事件(看下一章节，创建并注册事件)，你会想要libevent等待事件并在发生时提醒你。

默认的，event_base_loop()函数运行一个event_base直到没有更多的事件注册到里面。为了运行循环，它重复的检查是否有任何已注册的事件触发(例如，一个读事件的文件描述符准备好可读，或一个超时(timeout)事件准备好抛出异常)。一旦这些事件发生，它会将所有触发的事件设置为激活态并且开始运行他们。

你可以通过设置相应的标志位来改变event_base_loop的行为。

  • EVLOOP_ONCE 此标志位意味着循环将会等待，直到有一些事件处于激活态，然后调用回调函数运行激活的事件。直到没有更多的事件需要运行时，函数返回
  • EVLOOP_NONBLOCK 此标志位意味着，循环不会等待事件触发，仅会检查是否有任何事件准备好触发，如果触发就调用相应的回调函数执行
  • EVLOOP_NO_EXIT_ON_EMPTY 一般来说，当没有挂起和处于激活态的事件时，循环会退出。可以通过设置此标志位改变这种行为。例如，想要在其他线程中增加事件。如果设置了此标志位，循环将在调用event_base_loopbreak(), 或调用event_base_loopexit(),或发生错误时退出

终止一个循环

如果你想在所有的事件移除之前终止一个已经激活的事件循环，需要用到两个稍微有些不同的函数。

  • event_base_loopexit()函数告诉一个event_base在给定的时间后停止循环，如果参数tv是NULL，则事件循环不再延迟，立即停止。如果event_base正在运行回调函数处理一些激活的事件，则会继续运行他们直到全部执行完，并不会立即退出。
  • event_base_loopbreak()函数告诉event_base立即退出循环，和函数event_base_loopexit(base,NULL)不同，如果event_base正在运行某个激活事件的回调函数，它会在完成当前进程的任务后立即退出。

注意：当没有事件循环正在运行的时候，event_base_loopexit()和event_base_loopbreak()表现是不同的。loopexit调度事件循环的下一个实例，在下一个回调函数执行后立即退出；loopbreak在当前正在执行的事件循环中立即退出，当事件循环没有开始时，没有任何影响。

创建event_base实例

在你运用感兴趣的libevent函数进行开发之前，你需要分配一个或更多的event_base结构体。每一个event_base结构体管理一个事件集合并且通过轮询(poll)的方式决定那个事件处于活跃状态。如果event_base被设置为使用锁(locking),在多线程之间访问它是安全的。事件循环可以在单线程内进行循环，当然，如果你想利用多线程来轮询I/O事件，你需要在每一个线程中使用event_base结构体。每一个event_base有一个方法或后台程序，用来决定哪个事件已经准备好了。配置的方法与操作系统支持的方法有关，libevent对以下方法进行了打包：

  • win32

高性能服务器框架

Posted on 2019-05-22 | Edited on 2019-09-15

高性能服务器程序框架一般可以分为三个部分：

  • I/O处理单元
  • 逻辑单元
  • 存储单元

服务器模型

C/S模型

C/S模型逻辑上很简单。服务器启动后，首先创建一个或多个监听socket，并调用bind函数将其绑定到服务器感兴趣的端口上，然后调用listen函数等待客户连接。服务器稳定运行后，客户端就可以调用connect函数向服务器发起连接。由于客户端连接请求是随机到达的异步事件，服务器需要使用某种I/O模型来监听这一事件。I/O模型有很多种，select,epoll等都是。当监听到连接请求后，服务器就调用accept函数接受他，并分配一个新的逻辑单元为它服务。逻辑单元可以是新创建的子进程、子线程、或者其他。逻辑单元读取客户请求，处理该请求，然后将处理结果返回给客户端。一个典型流程如下图。

C/S模型适合资源相对集中的场合，并且它的实现也很简单，但其缺点也很明显：服务器是通信的中心，当访问量过大时，可能所有用户得到的响应都很慢。

P2P模型

P2P(peer to peer)点对点的模型摒弃了以服务器为中心的格局，让网络上所有主机重新回归对等的地位。模型使得每台主机消耗资源的同时也在为别人服务。当用户很多，连接量很大时，网络的负载将加重。

服务器模型编程框架

虽然服务器程序种类繁多，但基本框架都是一样的，不一样在于逻辑处理，基本的框架如下图所示。

服务器各模块描述
模块 单个服务器程序 服务器集群
I/O处理单元 处理客户连接，读写网络数据 作为接入服务器，实现负载均衡
逻辑处理单元 业务进程或线程 逻辑服务器

I/O处理单元

I/O处理单元是服务器管理客户连接的模块。通常完成以下工作：等待并接受新的客户连接，接受客户数据，将服务器响应数据返回给客户端。但是数据的收发不一定是在I/O处理单元中执行。

逻辑处理单元

一个逻辑处理单元通常是一个进程或线程。他分析并处理客户数据，然后将结果传递给i/o单元或直接发给客户端。

请求队列

请求队列是个单元之间的通信方式的抽象。I/O单元接收到客户请求时，需要以某种方式通知一个逻辑单元来处理请求。同样，各个逻辑单元同时访问一个存储单元时，也需要采用这种机制来协调处理静态条件。请求队列通常被视为池的一部分。

I/O模型

socket在创建的时候默认都是堵塞的，我们可以socket系统调用的第二个参数传递SOCK_NONBLOCK标志，或者通过fcntl设置为非阻塞的。

对于阻塞I/O，系统调用在执行时，可能因为无法立即返回而被操作系统暂时挂起，直到等待的事件发生为止。比如客户端向服务器发起连接的函数connect()，发送同步报文后，如果没有及时收到服务器发送的确认报文，则connect会被挂起。直到客户端收到确认报文并唤醒connect()调用。其他的accept,send,recv都是阻塞的。

对于非阻塞I/O执行的系统调用总是立即返回，而不管事件是否已经发生。如果事件没有立即发生，这些系统调用就返回-1,按照错误类型来处理。

显然，事件没发生时频繁唤醒I/O线程也会增加开销，只要在事件确定达到时唤醒I/O才是高效的，因此非阻塞I/O要和其他通知机制一起使用。比如I/O复用。

I/O复用是最常使用的I/O通知机制。应用程序通过I/O复用函数向内核注册一组事件，内核通过I/O复用函数把其中就绪的事件通知给应用程序。Linux上常用的I/O复用函数有select,poll,epoll.目前效率最高，使用最广泛的就是epoll机制，基于事件驱动的I/O复用。I/O复用函数本身是阻塞的，他们可以同时监听多个I/O的能力，并及时把准备的I/O返回给逻辑进行进一步处理。

需要注意的是，阻塞I/O, I/O复用，SIGIO都是同步IO。因为三种I/O的读写操作都是在I/O事件发生之后，由应用程序来完成的。

异步I/O的读写实质上是由内核来完成的，而不是在同步I/O中由应用程序来完成。异步I/O在读写时，不管是不是阻塞，数据由用户写给用户缓冲区后，系统内核在后台将数据由用户缓冲区写到内核缓冲区，I/O就绪后直接发送，发送完成后内核通知应用程序已经发送完成。

总结一下：

I/O模型 读写操作和阻塞阶段
阻塞I/O 程序阻塞于读写函数
I/O复用 程序阻塞于I/O复用函数，但同时监听多个I/O事件，I/O读写本身是非阻塞的
SIGIO 信号触发读写就绪事件，用户程序执行读写操作，没有阻塞阶段
异步I/O 内核执行读写操作并触发读写事件完成，没有阻塞阶段

高效的事件处理模式－Reactor

服务器通常需要处理三类事件:I/O事件、信号及定时事件。这里先介绍两种高效的事件处理模式:Reator和Proactor.随着网络设计模式的兴起，Reator和Proactor事件处理模式应允而生。同步I/O模型常用于实现Reactor模式。异步I/O模型常用来实现Preactor模式。主要介绍Reactor模式。

Reactor是这样一种模式，他要求主线程(I/O处理单元，下同)只负责监听文件描述符上是否有事件发生，有的话就立即将事件通知工作线程(逻辑单元，下同)。除此之外，主线程不做任何其他实质性的工作。读写数据、接受新的连接、以及处理客户请求均在工作线程中完成。

使用同步I/O模型，以epoll_wait()为例，实现的Reactor模式的工作流程是：

  1. 主线程在epoll内核事件表上注册socket上的读就绪事件
  2. 主线程调用epoll_wait()等待socket上的读就绪事件
  3. 当socket上有数据可读时，epoll_wait()通知主线程。主线程将socket可读事件放入请求队列
  4. 睡眠在请求队列上的某个工作线程被唤醒，他从socket读取数据并处理客户请求，然后向epoll内核事件表中注册socket上的写就绪事件
  5. 主线程调用epoll_wait()等待socket可写
  6. 当socket可写时，epoll_wait()通知主线程，主线程将socket可写事件放入请求队列
  7. 睡眠在请求队列上的某个工作线程被唤醒，它从socket上写服务器处理客户请求的结果

工作线程从请求队列中读取事件后，将根据事件的类型来决定如何处理它：对于可读事件，执行读数据和处理请求的操作；对于可写事件，执行写数据的操作。因此上图中的工作线程是没有必要区分所谓的”读”“写”线程。

Reactor模式框架

使用Reactor框架，必备的几个组件：事件源，Reactor框架，多路复用机制和事件处理程序。Reactor模型的整体框架如下图所示。

  • 事件源 ：linux上指的是文件描述符
  • event demultiplexer 事件多路分发机制 : 由操作系统提供的io多路复用机制，比如select和epoll。
    • 程序首先将其关心的事件源及其事件注册到event demultipexer上;
    • 当有事件到达时，event demultipelexer会发出通知在”已注册的句柄集中，一个或多个句柄的事件已经就绪”;
    • 程序收到通知后，就可以在非阻塞的情况下对事件进行处理了。
  • Reactor反应器 : Reactor是事件管理的接口，内部使用event demultilexer注册，注销事件，并运行事件循环，当有事件进入”就绪”态时，调用注册事件的回调函数进行处理。

两种高效的并发模式

从实际上来说，并发编程主要有多进程和多线程两种方式，这一部分先讨论并发模式。并发模式是指I/O处理单元和多个逻辑单元之间协调完成任务的方法，服务器有两种并发编程模式：半同步/半异步模式和领导者/追随者模式。

半同步/半异步模式

首先，半同步/半异步模式中的同步和异步与前面讨论的I/O模型中的同步和异步是完全不同的概念。在I/O模型中，同步和异步指的是内核向应用程序通知的是何种I/O事件(就绪事件还是完成事件)，以及该由谁来完成I/O读写(应用程序还是内核)。　在并发模式中，_同步_指的是程序完全按照代码序列的顺序执行；_异步_指的是程序的执行需要事件来驱动。常见的系统事件包括中断、信号等。

同步模式实时性差，效率低，但逻辑简单；异步实时性高，执行效率高，但程序复杂，难于调试和扩展。在服务器这种既需要实时性又需要效率的地方，就需要同时使用两种模式。在同步/异步模式中，同步模式用于处理客户逻辑逻辑，对应于逻辑单元；异步模式用于处理I/O事件。异步线程接收到客户请求后，就将其封装成请求对象并插入请求队列中。请求队列将通知某个工作线程在同步模式的工作线程读取并处理该请求。如何选择工作线程取决于请求队列的设计

领导者/追随者模式

领导者/追随者模式是多个线程轮流获得事件源集合，轮流监听、分发并处理事件的一种模式。在任意时间点，程序都仅有一个领导者,他负责监听I/O事件。而其他的线程都是追随者，他们休眠在线程池中等待成为新的领导者。当前的领导者如果检测到I/O事件，首先要从线程池中推选出新的领导线程，然后处理I/O事件。此时，新的领导者等待新的I/O事件，而原来的领导者则处理I/O事件，二者实现了并发。

领导者/追随者模式包含如下几个组件:句柄集，线程集，事件处理器和具体的事件处理器。

  • 句柄集：用于表示I/O资源，在linux就是文件描述符。
  • 线程集:所有工作线程的管理者，他负责各线程之间的同步，以及新领导者线程的推选。线程集中的线程在任意时刻都处于以下三种状态中：
    • leader 线程处于领导者身份，负责等待句柄上集上的I/O事件。
    • processing 线程正在处理事件。领导检测到I/O事件后，可以转移到processing状态来处理该事件。
    • follower 线程当前处于追随者身份

有限状态机

前面介绍了服务器的I/O单元、请求队列和逻辑单元之间协调完成任务的各种模式，这一部分讨论逻辑单元内部一种高效编程方法:有限状态机。有的应用层协议头部包含数据包类型字段，每种类型可以映射为逻辑单元的一种执行状态，服务器可以根据它来编写相应的处理逻辑。

2019-5-22-tcp协议详解.md

Posted on 2019-05-22 | Edited on 2019-09-20

TCP协议是TCP/IP协议族中另一个重要的协议。和IP协议相比，TCP协议更接近应用层，有更强的操作性。一些重要的socket选项都和TCP协议相关。本节从以下4个方面讨论TCP协议。

  • TCP头部信息
  • TCP状态转移过程：从连接到断开的整个过程，状态的迁移
  • TCP数据流
  • TCP数据流的控制：超时重传和拥塞控制

TCP服务的特点

TCP相对于UDP协议的特点是：面向连接，字节流，可靠传输。　

面向连接

使用TCP协议通信的双方必须建立连接，然后再开始数据的读写。双方都必须为该TCP链接分配必要的内核资源，以管理连接状态和连接上的数据传输。TCP连接是全双工的，双方的数据通信可以在一个连接中完成。完成数据交换后，通信双反都必须断开连接以释放系统资源。TCP协议的连接是一对一的，所以基于广播和多播的应用程序不能使用TCP服务。无连接的UDP协议非常适合广播和多播。

TCP字节流服务和UDP数据报服务

字节流服务和数据报服务的区别对应到实际编程中体现为通信双方是否执行相同次数的读写操作。在TCP连接中，发送端连续执行多次发送操作时，数据先放入TCP缓冲区中，当TCP真正开始发送数据时，发送缓冲区中这些等待发送的数据可能被封装成一个或多个TCP报文段发送。因此，TCP发送出的报文段的数量和应用程序执行发送操作的次数没有固定数量关系。

可靠传输

  • TCP传输是可靠的。TCP首先采用发送应答机制，即发送端发送的每个TCP报文段都必须得到接收方的应答，才认为TCP报文传输成功。
  • 其次，TCP协议采用超时重传机制，发送端发送完一个TCP报文段后启动定时器，如果在定时器内未收到应答，他将重传该报文段。
  • 最后，TCP数据报是以IP数据报发送的，而IP数据报到达接收端可能乱序，重复，所以TCP协议还会对接受到的TCP报文段进行重排、整理，再交给应用层。
  • TCP提供流量控制。TCP总是告诉对端在任何时刻它一次能够从对端接受多少字节的数据，这称之为通告窗口。在任何时刻，该窗口指出接收缓冲区中当前可以用的空间量，从而确保发送端发送的数据不会使接收端缓冲区溢出
  • TCP连接是全双工的，TCP必须为每个数据流方向上跟踪诸如序列号和通告窗口大小等状态。

TCP头部

TCP头部的结构如下所示

比较重要的几个部分作如下解释：

  • 32位序列号：一次TCP通信中，某个传输方向上的字节流的每个字节的编号。
  • 32位确认号：用作对另一方发送的数据的响应。其值是收到的TCP报文段的序号+1。主机A和主机B通信，则A发送给B的TCP报文段不仅携带自己的序号，也包含对B发送来的TCP的报文段的确认号。反之依然。
  • 16位窗口大小:流量控制的一种手段。接受通告窗口，告诉对方本端的TCP缓冲区还能容纳多少字节的数据，这样对方可以控制发送数据的速度。

TCP连接建立和关闭

从两个方面同时理解TCP连接的建立和终止过程：报文的状态转移，socket函数编程。

在上图的建立和断开时序图中，左边的客户端主动发起建立连接的请求，对应于connect()函数，已经绑定好套接字端口；右边的服务端开启listen()监听连接请求。客户端发送在connect()的过程中，先发送同步报文段seq1, 服务端收到SYN字段的报文后，由操作系统内核将其插入SYＮ队列中，并向客户端发送SYN和ACK字段报文。客户端收到服务端发送的同步和确认字段后，发送确认字段，值为服务端发送的序列字段值+1。服务端内核收到客户端发来的确认字段后，将对应的连接由SYN队列转入ACCPET队列，再有accept()函数取出ACCEPT队列中的第一个套接字进行通信。SYN队列和ACCPET队列的长度与listen()函数中设置的backlog的值有关。

当客户端执行主动关闭时，他将向服务器发送一个结束报文段，同时进入FIN_WAIT_1状态。此时客户端若收到服务端对结束报文段的确认字段ACK，则进入FIN－WAIT_2状态。当客户端处于FIN－WAIT_2状态时，服务器处于CLOSE_WAIT状态。此时服务端也关闭连接，发送结束报文段FIN，则客户端也将进行确认ACK，此时客户端进入TIME_WAIT状态。处于FIN_WAIT_2的客户端需要收到服务端发送的结束报文段FIN才能进入TIME_WAIT状态，否则将一直停留在在这个状态。如果不是为了在半关闭状态下继续接受数据，连接长时间停留在FIN－WAIT_2没有意义。连接停留在FIN—WAIT_2状态可能发生在：客户端执行半关闭后，没等服务端发送结束报文段便强行终止客户端。此时客户端的连接交由系统内核来接管，可称之为孤儿连接。系统内核定义了能接管的孤儿连接数目和孤儿连接可以在系统内核中存在的生存时间来管理孤儿连接。

为什么要有TIME_WAIT状态？MSL是任何IP数据报在互联网中存活的最长时间

从上图来看，客户端接受服务端的结束报文段并发送确认报文段后，并没有进入CLOSE状态，而是进入TIME_WAIT状态，原因如下：

  • 可靠的终止TCP连接
    • 因为服务端可能没有收到ack报文，将重新给客户端发送FIN报文，需要客户端重新发送最后的ACK
  • 保证让迟来的TCP报文段有足够时间识别并丢弃
    • TCP必须防止来自某个连接的老的重复分组在该链接已终止后再现，从而被误解为属于同一连接的某个新的化身。为了做到这一点，TCP将不给处于TIME-WAIT状态的连接发起新的化身。既然TIME_WAIT状态的时间是MSL的2倍，这就足以让某个方向上的分组最多存活MSL秒就被抛弃，另一个方向上的应答最多存活MSL秒也被丢弃。通过实施这个规则，我们就能保证没成功建立一个TCP连接，来自该连接先前化身的老的重复分组都已在网络中中断了

2019-5-13-I/O复用:select-poll-epoll.md

Posted on 2019-05-13 | Edited on 2019-09-15

总结几种IO复用的方法:select,poll,epoll。内容来自《unix网络编程卷１套接字联网API》以及几篇博客的整理和讲解。

select，poll，epoll都是IO多路复用的机制。I/O多路复用就通过一种机制，可以监视多个描述符，一旦某个描述符就绪（一般是读就绪或者写就绪），能够通知程序进行相应的读写操作。但select，poll，epoll本质上都是同步I/O，因为他们都需要在读写事件就绪后自己负责进行读写，也就是说这个读写过程是阻塞的，而异步I/O则无需自己负责进行读写，异步I/O的实现会负责把数据从内核拷贝到用户空间。

IO模型

IO复用典型使用在下列的网络应用场合：

  • 当客户处理多个描述符时(通常是交互式输入和网络套接字)，必须使用io复用。
  • 一个客户处理多个套接字是可能的。
  • 一个TCP服务器既要处理监听套接字又要处理已连接套接字，一般就要使用io复用。
  • 一个服务器同时处理tcp.udp两种协议，就要使用io复用。
  • 一个服务器处理多个服务协议或者多个协议。

io复用不止运用于网络编程，许多重要的应用也会用到这项技术。与多进程和多线程技术相比，I/O多路复用技术的最大优势是系统开销小，系统不必创建进程/线程，也不必维护这些进程/线程，从而大大减小了系统的开销。

在介绍具体的io复用技术之前先介绍unix可用的5种可用的io模型的基本区别:

  • 阻塞式io
  • 非阻塞式io
  • io复用(select,poll,epoll)
  • 信号驱动式io
  • 异步io

前四种都是同步io，只有最后一个是异步io。

阻塞式io

最流行的io模型是阻塞式io模型。默认情况下，所有的套接字都是阻塞的。

非阻塞式io

进程把一个套接字设置成非阻塞式io时，是在通知内核：当所请求的i/o操作非得把本进程投入睡眠才能完成时，不要把本进程投入睡眠，而是返回一个错误。当一个应用进程像这样对一个非阻塞式描述循环调用读操作时，我们称之为轮询(polling)。应用进程持续轮询内核，已查看某个操作是否就绪。

在两次轮询的间隙，应用进程是否可以去做别的工作？

io复用模型

有了io复用，我们就可以调用select或poll，阻塞在这两个系统调用中的某一个之上，而不是阻塞在真正的io系统调用之上。

信号驱动模型

内核在描述符就绪时通知应用进程，这种模型为信号驱动式io。

select函数

该函数允许进程指示内核等待多个事件中的任何一个发生，并只在有一个或多个事件发生或经历一段指定的时间后才唤醒它。

struct fd_set可以理解为一个集合，这个集合中存放的是文件描述符，可通过以下四个宏进行设置：

  • FD_ZERO(&fd_set) //清空集合
  • FD_SET(int fd, fd_set *fdset) //将给定的文件描述符fd加入集合中
  • FD_CLR(int fd, fd_set *fdset) //将一个给定的文件描述符从集合中删除
  • FD_ISSET(int fd, fd_set *fdset) //检查集合中指定的文件描述符是否可以读写

poll

poll的机制与select类似，与select在本质上没有多大差别，管理多个描述符也是进行轮询，根据描述符的状态进行处理，但是poll没有最大文件描述符数量的限制。poll和select同样存在一个缺点就是，包含大量文件描述符的数组被整体复制于用户态和内核的地址空间之间，而不论这些文件描述符是否就绪，它的开销随着文件描述符数量的增加而线性增大。

每一个pollfd结构体指定了一个被监视的文件描述符，可以传递多个结构体，指示poll()监视多个文件描述符。每个结构体的events域是监视该文件描述符的事件掩码，由用户来设置这个域。revents域是文件描述符的操作结果事件掩码，内核在调用返回时设置这个域。events域中请求的任何事件都可能在revents域中返回。合法的事件如下：

该图可以分为四个部分，第一部分处理输入的四个常值；第二部分是处理输出的三个常值；第三个部分是处理错误的三个常值。其中第三个部分的三个常值不能在events中设置，但是相应条件存在时就在revents中返回

就tcp/udp通信而言，一下条件引起poll返回特定的revents。

  • 所有正规TCP数据和所有UDP数据都被认为是普通数据。
  • TCP的带外数据被认为是优先级带数据。
  • 当TCP连接的读半部关闭时，例如收到对端的FIN，也被认为是普通数据，随后的读操作将返回0。
  • TCP连接存在错误也被认为是普通数据。
  • 在监听套接字上有新的连接可用的时候既可以认为是普通数据，也可以认为是优先级数据。大多数情况下是普通数据。
  • 非阻塞式connect的完成被认为是使用相应套接字可写。

epoll

epoll是之前的select和poll的增强版本。相对于select和poll来说，epoll更加灵活，没有描述符限制。epoll使用一个文件描述符管理多个描述符，将用户关系的文件描述符的事件存放到内核的一个事件表中，这样在用户空间和内核空间的copy只需一次。

epoll接口

epoll操作过程需要三个接口，分别如下

1. int epoll_create(int size)

创建一个epoll的句柄，size用来告诉内核这个监听的数目一共有多大。这个参数不同于select()中的第一个参数，给出最大监听的fd+1的值。需要注意的是，当创建好epoll句柄后，它就是会占用一个fd值，在linux下如果查看/proc/进程id/fd/，是能够看到这个fd的，所以在使用完epoll后，必须调用close()关闭，否则可能导致fd被耗尽。

2. int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event)

epoll的事件注册函数，它不同与select()是在监听事件时告诉内核要监听什么类型的事件epoll的事件注册函数，它不同与select()是在监听事件时告诉内核要监听什么类型的事件，而是在这里先注册要监听的事件类型。第一个参数是epoll_create()的返回值，第二个参数表示动作，用三个宏来表示：

  • EPOLL_CTL_ADD：注册新的fd到epfd中；
  • EPOLL_CTL_MOD：修改已经注册的fd的监听事件；
  • EPOLL_CTL_DEL：从epfd中删除一个fd；
    第三个参数是需要监听的fd，第四个参数是告诉内核需要监听什么事，struct epoll_event结构如下：

events可以是以下几个宏的集合：

  • EPOLLIN ：表示对应的文件描述符可以读（包括对端SOCKET正常关闭）；
  • EPOLLOUT：表示对应的文件描述符可以写；
  • EPOLLPRI：表示对应的文件描述符有紧急的数据可读（这里应该表示有带外数据到来）；
  • EPOLLERR：表示对应的文件描述符发生错误；
  • EPOLLHUP：表示对应的文件描述符被挂断；
  • EPOLLET： 将EPOLL设为边缘触发(Edge Triggered)模式，这是相对于水平触发(Level Triggered)来说的。
  • EPOLLONESHOT：只监听一次事件，当监听完这次事件之后，如果还需要继续监听这个socket的话，需要再次把这个socket加入到EPOLL队列里

3. epoll_wait(int epfd, struct epoll_event * events, int maxevents, int timeout)

等待事件的产生，类似于select()调用。参数events用来从内核得到事件的集合，maxevents告之内核这个events有多大，这个maxevents的值不能大于创建epoll_create()时的size，参数timeout是超时时间（毫秒，0会立即返回，-1将不确定，也有说法说是永久阻塞）。该函数返回需要处理的事件数目，如返回0表示已超时。

工作模式

　　epoll对文件描述符的操作有两种模式：LT（level trigger）和ET（edge trigger）。LT模式是默认模式，LT模式与ET模式的区别如下：

　　LT模式：当epoll_wait检测到描述符事件发生并将此事件通知应用程序，应用程序可以不立即处理该事件。下次调用epoll_wait时，会再次响应应用程序并通知此事件。

　　ET模式：当epoll_wait检测到描述符事件发生并将此事件通知应用程序，应用程序必须立即处理该事件。如果不处理，下次调用epoll_wait时，不会再次响应应用程序并通知此事件。

　　ET模式在很大程度上减少了epoll事件被重复触发的次数，因此效率要比LT模式高。epoll工作在ET模式的时候，必须使用非阻塞套接口，以避免由于一个文件句柄的阻塞读/阻塞写操作把处理多个文件描述符的任务饿死
